# 背景
Finatextでバックエンドエンジニアをしているデニスです。普段は「Inspire」という保険ビジネスプラットフォームを開発していますが、半年ほど前に、様々なEnabling Teamが新しく結成されました。[Enabling Team](https://techblog.finatext.com/enabling-team-6938b9135ddc)というのは、「今のプロダクト開発の直線的なマイルストーンには乗らない技術を検証し、習得したりノウハウを溜めた上で、これを実プロダクト・実環境に導入するサポートをおこなう」チームのことです。中でも私はInsurtechチームの代表として、「自動推論」チームに携わっています。

自動推論チームは、社内で開発している証券サービス、貸金サービス、保険サービスの実装を自動推論によってより安定させるための調査・研究を行っています。それぞれのサービスのバックエンドが全てGoで書かれているため、自動推論の対象はGoのみとなっています。そのため、例えばRustの世界における[kani](https://github.com/model-checking/kani)というツールにあたるものをGoの世界で探しています。

この記事では自動推論チームで調べたことについて紹介したいと思います。

# 先行研究

Kickoff後に、全メンバーが「自動推論ってなんぞや」というレベルの知識だったので、ひとまず各自が調査に入りました。私は以前から少し静的解析に趣味があったため、その観点で先行研究を行いました。

## "それっぽい"ツール

#### Gomela
https://github.com/nicolasdilley/Gomela
Go言語のソースコードから**並行処理**の振る舞いモデル（Promelaモデル）を自動的に抽出し、SPINモデルチェッカーで検証する有界モデル検査ツール。

#### GCatch
https://github.com/system-pclub/GCatch
**並行処理**に起因するバグを静的に検出し、自動修正（GFix）もできるツールスイート。

#### godel-checker
https://bitbucket.org/MobilityReadingGroup/godel-checker/src/master/
MiGo型を用いて**並行処理**の正しさをモデル検査で検証するためのツール。

GobraはViperプロジェクトの中でGo言語に対応するフロントエンドのことです。

#### それぞれのコンポーネントの概要

###### Gobra
GoコードとそのアノテーションをSilverという言語に変換するフロントエンドです。
https://github.com/viperproject/gobra

###### Silver
中間検証言語 (Intermediate Verification Language, IVL)です。
https://github.com/viperproject/silver

###### Viperserver
SilverもしくはSilverのASTを受け取り、CarbonもしくはSiliconに推論を依頼するHTTP兼LSPサーバーです。
ラッパー・オーケストレーターのような立ち位置です。
https://github.com/viperproject/viperserver

###### Carbon
検証条件生成 (Verification Condition Generation, VCG) のバックエンド。
Z3というソルバーに推論を依頼するために、Viper専用の中間言語であるSilverを、Boogieに変換する必要があります。BoogieはMicrosoft Researchが開発した汎用の中間検証言語です。
https://github.com/viperproject/carbon

###### Silicon
シンボリック実行検証 (Symbolic Execution Verification) のバックエンド。
SiliconのほうがCarbonより新しいバックエンドです。基本的にCarbonは使わないで問題ありません。
Z3の他にcvc5という別のソルバーも対応し、ソルバーを内部で呼び出すことができます。
https://github.com/viperproject/silicon

###### Z3
SMTソルバーです。
SMTはSatisfiability Modulo Theories（背景理論付き充足可能性判定問題）の略です。
https://github.com/Z3Prover/z3

###### cvc5
SMTソルバーです。
https://github.com/cvc5/cvc5

# Gobra でできること

Gobraは、Goにいくつかのアノテーションを加えることで、簡単な推論を行うことができます。

#### assert

`assert`句は最も単純なアノテーションです。書かれている時点で、ある条件が満たされていなければエラーになるというアノテーションです。
使い方は以下の通りです。

```go
func AbsPlusOne(x int) (res int) {
	if x >= 0 {
		res = x + 1
		// @ assert res >= 1
	} else {
		res = -x + 1
		// @ assert res >= 2
	}
	return
}
```
テストなどでもよく使われる一般的なアサーションです。以下の場合にfailします。

```go
func Double_Fail(n int) (res int) {
	res = 2 * n
	// @ assert res > n
	return
}
```

#### ensures

`ensures`句は関数終了時の保証として使うことができます。関数が正常にリターンするとき（パニックや無限ループに陥らない場合）に必ず真でなければならないと主張するアサーションです (Postcondition)。
特徴としては、返却値を認識するために基本的に名前付き返却値にする必要があります。

使い方は以下の通りです。

```go
// @ ensures res >= 0 && (res == x || res == -x)
func Abs_OK(x int) (res int) {
	if x < 0 {
		return -x
	} else {
		return x
	}
}
```
例えば次のような実装ミスがあった場合は、`ensures`が失敗します。
```go
// @ ensures res >= 0 && (res == x || res == -x)
func Abs_Fail(x int) (res int) {
	if x < 0 {
		return x // 実装ミス
	} else {
		return x
	}
}
```
もちろん、複数回使うことも可能です。
```go
// @ ensures res1 != res2
// @ ensures res1 == a
// @ ensures res2 == -a
func Multiple_Fail1(a int) (res1, res2 int) {
	return a, -a
}
```
こちらの場合は、最初の`ensures`のほうが失敗し、2つ目、3つ目が成功します。

`ensures`はインターフェースも対応していますが、使い方が少し複雑です。最初は以下のような書き方で実装を全て検知できると思いました。

```go
type Formatter interface {
	// @ ensures len(s) > 0
	Format() (s string)
}

var _ Formatter = a{}
type a struct{}

func (a) Format() (s string) {
	return "a"
}
```

が、Gobraの仕様としては、インターフェースにアノテーションが入った場合、実装側にも必須になります。
すなわち、上記のコードも以下のコードも失敗します。なぜ以下のほうが失敗するかと言いますと、実装側の`ensures`はインターフェース側の`ensures`より"ゆるい"からです。
最低でもインターフェースにあるほうの`ensures`を満たさなければなりません。

```go
// @ ensures len(s) >= 0
func (a) Format() (s string) {
	return "a"
}
```

正しくは以下ですね。


```go
// @ ensures len(s) == 1
func (a) Format() (s string) {
	return "a"
}
```

#### requires

`requires`句は前提条件を表すアノテーションです。関数が呼び出される前に満たされていなければならない条件を表現します (Precondition)。
前提条件のアサーションが満たされていない場合は、呼び出し元でエラーが発生します。

使い方は以下の通りです。

```go
// @ requires a > 0
// @ ensures res > 0
func OK(a int) (res int) {
	return a
}

func _() {
	_ = OK(10)  // 成功
	_ = OK(-10) // 失敗
}
```

#### invariant

`invariant`句はループ不変条件を表します。ループの各イテレーションの開始時と終了後に満たされていなければならない条件を定義することができます (Loop Invariant)。

使い方は以下の通りです。

```go
// @ requires n >= 0
// @ ensures sum == (n * (n+1)) / 2
func SumUpTo_OK(n int) (sum int) {
	sum = 0

	// @ invariant i >= 0 && i <= n+1
	// @ invariant sum == i * (i-1) / 2
	for i := 0; i <= n; i++ {
		sum += i
	}

	return sum
}
```

ほとんど`assert`と同じ仕様ですが、ソルバーがコードを実行しているわけではないので、ループの中に`assert`を書いてもループが0回実行されるのか、1億回実行されるのか、無限ループになるのかはコードを見るだけではわかりません。そのため、ループの中の処理を証明するには数学的帰納法のような方法で証明する必要があります。

#### trusted

`trusted`句は関数を証明済みとしてみなします。単純に「この関数は絶対正しい」というとき、もしくはアノテーションを書くのが困難なときに便利です。

使い方は以下の通りです。

```go
// @ ensures res < 0
// @ trusted
func One() (res int) {
	return 1
}
```

どう見ても`ensures`が失敗するはずですが、`trusted`があることで正しいとみなされます。

## アノテーションが書けない場合

ここまでの流れだと、自分が書いているコードにアノテーションを付ければGobraが極力推論しようとしてくれますが、アノテーションを書けないコードもありますよね？標準ライブラリやインポートしたパッケージの中のコードはどうしたら良いでしょう？

実は、Gobraを騙す方法はあります（※仕様です）。

以下のような"ダミー形式"で例えば`fmt.Stringer`に`ensures`をつけることができます。

`./fmt/dummy.go`
```go
package fmt

type Stringer interface {
	// @ ensures len(s) > 0
	String() (s string)
}
```
`./main.go`
```go
package main

import (
	"fmt"
)

var _ fmt.Stringer = a{}
type a struct{}

// @ ensures len(s) == 1
func (a) String() (s string) {
	return "a"
}
```

これでGobraの取り入れハードルがだいぶ上がりますね⋯⋯

## Gobra でできないこと

1つの弱点としては、Gobraは数値型を全て数学的な整数（unbounded integer）として扱うため、`uint`が必然的に非負であることを認識できません。そのため、次のような`ensures`は失敗してしまいます。

```go
// @ ensures res >= 0
func Fail(a int) (res uint) {
	return uint(a)
}
```

数学的に不可能であっても、明示的に書く必要があります。次のケースは問題なく通ります。

```go
// @ ensures res >= 0
func OK(a int) (res uint) {
	res = uint(a)
	if res < 0 {
		return 0
	}
	return res
}
```

# まとめ
率直に言えば、Gobraに関しては、バックエンドでの汎用的な使用はまだ難しいと思っています。GobraのアノテーションをLLMやLinterで自動的に生成することも試みましたが、引っかかるところが多く、コンパイルエラーの発生率も高かったです。やはり手動でアノテーションをつけるしかないと考えています。

一方で、PoCもしくは設計を考える段階では、Gobraで確認しておくのはメリットがあるかもしれません。

自動推論のEnabling Teamは引き続き稼働中ですし、Gobraの開発もまだ継続されているので、引き続きウォッチしていこうと思っています。
